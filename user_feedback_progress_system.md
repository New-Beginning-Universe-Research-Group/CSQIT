# 用户体验改进建议：复杂任务的进度可视化与控制机制

## 背景

在执行复杂任务（如代码编辑、编译、大规模数据处理）时，用户经常遇到界面"无反应"的情况，无法判断是任务卡死了还是在正常运行中。这严重影响用户体验和信任度。

此外，用户可能在多个终端/设备上同时使用同一对话，但任务进度无法同步，导致重复操作或信息不一致。

更关键的问题是：当任务执行时，终端输出往往会淹没对话框，使用户无法与 AI 进行沟通，用户的问题和建议无法及时传达。

## 需求概述

**目标**：
1. 为长时间运行的任务提供实时进度反馈和基本控制机制
2. 确保同一对话在所有终端/设备上同步显示任务进度
3. **保持对话框的最高优先级，确保用户始终可以与 AI 沟通**

---

## 功能需求

### 一、任务进度可视化与控制

#### 1. 任务启动时的明确预期
- 启动时显示预估耗时范围（如"预计 2-5 分钟"）
- 明确说明会如何展示进度（如进度条、定期汇报等）
- 关键里程碑提示（如"正在编译核心模块..."）

#### 2. 执行中的实时反馈
- **进度条**：可视化完成百分比
- **阶段性汇报**：定期（如每10%）输出关键状态信息
- **中间结果**：对于有中间结果的任务，显示当前计算结果
- **预估剩余时间**：根据当前进度估算剩余时间

#### 3. 基本控制机制
- **暂停/恢复**：支持快捷键暂停任务
- **检查点保存**：对于可中断的任务，支持从中断点恢复
- **终止确认**：用户想终止时，提供确认和清理机制

#### 4. 统一的反馈格式
```
[开始] 📦 任务名称
[预估] ⏱️  预计耗时：X-Y 分钟
[进度] ████████░░░░░░░░ 40% (2/5 步)
[状态] 正在执行：xxx模块编译中
[时间] 已用时 2分钟，预计剩余 3分钟
```

### 二、多终端同步功能

#### 1. 核心需求
- **同一对话，同步显示**：当一个终端启动任务后，其他所有打开同一对话的终端应同步显示相同的进度
- **实时状态同步**：任务状态（如暂停、恢复、完成）在所有终端保持一致
- **双向同步**：任何终端的操作（如暂停、终止）应实时反映到所有终端

#### 2. 同步场景
| 场景 | 当前行为 | 期望行为 |
|------|----------|----------|
| 终端A启动任务 | 终端B无显示 | 终端B自动显示进度条 |
| 终端A暂停任务 | 终端B仍显示运行中 | 终端B同步显示暂停状态 |
| 终端B终止任务 | 终端A继续运行 | 终端A同步显示已终止 |
| 任务完成通知 | 仅终端A收到 | 所有终端都收到通知 |

#### 3. 技术要求
- **低延迟同步**：状态变化应在 1 秒内同步到所有终端
- **离线容错**：短暂断网后重连应自动恢复同步
- **状态持久化**：刷新页面或切换设备后应恢复当前任务状态

### 三、对话框优先级保障

#### 1. 核心问题
当前问题：任务执行时，终端输出（如编译日志）往往会淹没对话框，用户无法：
- 询问任务状态
- 提出新的问题或修改需求
- 报告遇到的问题
- 取消或调整当前任务

#### 2. 解决方案
- **对话框置顶显示**：无论任务输出多少，对话框始终保持在最上层
- **独立的消息通道**：AI 的消息与任务输出分离，互不干扰
- **消息通知机制**：当有新消息时，有明确的视觉提示（如闪烁、角标）
- **静默模式选项**：用户可以选择将任务输出折叠或最小化，专注于对话

#### 3. 交互设计建议
```
┌─────────────────────────────────────────┐
│  [进度条：████████░░░░░░░ 40%]          │
│  正在编译：Core.lean                    │
├─────────────────────────────────────────┤
│  📝 对话框（始终可见）                   │
│  ┌─────────────────────────────────┐    │
│  │ 用户：编译卡住了怎么办？          │    │
│  │ AI：  编译时间较长，请稍候...    │    │
│  │       当前进度 40%，预计还需 3分钟│    │
│  └─────────────────────────────────┘    │
│  [输入框]                        [发送] │
├─────────────────────────────────────────┤
│  📄 任务输出（可折叠）                   │
│  > lake build ...                       │
│  > compiling Core.lean                  │
│  > ... (200+ 行)                        │
└─────────────────────────────────────────┘
```

#### 4. 技术实现要点
- **分层渲染**：对话框使用独立的 UI 层，高于任务输出层
- **消息队列**：AI 消息进入独立队列，用户可随时查看
- **可折叠输出**：任务输出默认折叠，用户可展开查看详细日志
- **通知系统**：即使在输出展开时，新的 AI 消息也应出现通知提示

## 适用场景

- 代码编译（lake build, make, cargo build 等）
- 大规模数据处理/分析
- 文件批量操作
- 复杂搜索/替换任务
- 测试套件运行
- 模型训练/推理
- **跨设备协作**：用户在家用电脑和公司电脑同时使用同一对话
- **实时沟通**：任务执行中随时可能有问题或修改需求

---

## 用户价值

1. **消除焦虑**：用户清楚知道任务状态，不再担心是否卡死
2. **提高效率**：可以根据预估时间安排工作
3. **可控性**：支持暂停恢复，适应被打断的工作场景
4. **透明度**：中间结果可见，便于及时发现问题
5. **多设备一致性**：无需盯着同一设备，在任何设备上都能掌握进度
6. **沟通无阻**：任务执行中也能与 AI 保持沟通，反馈问题或调整需求

## 优先级建议

### P0（必须实现）
- **对话框优先级保障** ⭐（新增，核心交互保障）
- 进度百分比显示
- 定期状态汇报
- 预估耗时提示
- 多终端进度同步

### P1（重要）
- 检查点保存/恢复
- 暂停/恢复控制
- 中间结果可视化
- 可折叠的任务输出面板

### P2（增强）
- 终止确认与清理
- 离线容错机制
- 自定义反馈详细程度
- 消息通知系统

---

## 技术实现建议

1. **标准化接口**：提供统一的任务进度回调接口，供不同任务类型使用
2. **WebSocket 实时通信**：用于多终端状态同步
3. **状态持久化**：使用数据库或分布式缓存存储任务状态
4. **客户端轮询/推送混合**：无 WebSocket 支持时降级为轮询
5. **配置项**：提供用户配置选项，允许关闭多终端同步（针对隐私敏感场景）
6. **分层 UI 架构**：对话框独立于任务输出层，确保始终可交互
7. **消息异步队列**：AI 消息通过独立通道传输，不受任务输出阻塞
